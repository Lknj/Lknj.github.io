测试用例的编写从以下几个方面考虑会更全面：
  1. 考虑历史数据
  2. 考虑版本兼容问题 和 手机兼容
  3. 考虑 数据同时、多次、大量写入问题
  4. 考虑服务依赖问题
  5. 考虑 数据安全问题
  6. 模块被复用，测试被复用模块的原有功能使用情况
  7. 产品的可用性
  8. 产品的移植性
  9. 排序问题，相同数据根据什么排序
  10. 数据句状态问题，例如： 认证 已认证 认证中
实际案例
登陆功能用例

功能测试点

  1. 输入已注册的用户名和正确的密码，验证是否登录成功；

  2. 输入已注册的用户名和不正确的密码，验证是否登录失败，并且提示信息正确；

  3. 输入未注册的用户名和任意密码，验证是否登录失败，并且提示信息正确；

  4. 用户名和密码两者都为空，验证是否登录失败，并且提示信息正确；

  5. 用户名和密码两者之一为空，验证是否登录失败，并且提示信息正确；

  6. 如果登录功能启用了验证码功能，在用户名和密码正确的前提下，输入正确的验证码，验证是否登录成功；

  7. 如果登录功能启用了验证码功能，在用户名和密码正确的前提下，输入错误的验证码，验证是否登录失败，并且提示信息正确

  8. 输入正确的用户名和密码登录成功

  9. 输入错误的用户名密码登录失败

  10. 用户名正确，密码错误，是否提示输入密码错误？

  11. 用户名错误，密码正常，是否提示输入用户名错误？

  12. 用户名和密码都错误，是否有相应提示？

  13. 用户名密码为空时，是否有相应提示？

  14. 如果用户未注册，提示请先注册，然后进行登录

  15. 已经注销的用户登录失败，提示信息友好？

  16. 密码框是否加密显示？

  17. 用户名是否支持中文、特殊字符？

  18. 用户名是否有长度限制？

  19. 密码是否支持中文，特殊字符？

  20. 密码是否有长度限制？

  21. 密码是否区分大小写？

  22. 密码为一些简单常用字符串时，是否提示修改？如：123456

  23. 密码存储方式？是否加密？

  24. 登录功能是否需要输入验证码？
    ·验证码有效时间？
    ·验证码输入错误，登录失败，提示信息是否友好？
    ·输入过期的验证能否登录成功？
    ·验证码是否容易识别？
    ·验证码换一张功能是否可用？点击验证码图片是否可以更换验证码？

  25. 用户体系：比如系统分普通用户、高级用户，不同用户登录系统后可的权限不同。

  26. 如果使用第三方账号 (QQ,微博账号) 登录，那么第三方账号与本系统的账号体系对应关系如何保存？首次登录需要极权等。
    安全测试

  27. 不登录：浏览器中直接输入登录后的地址，看是否可以直接进入

  28. 登录成功后生成的 Cookie，是否是 httponly(否则容易被脚本盗取)

  29. 用户名和密码是否通过加密的方式，发送给 Web 服务器

  30. 用户名和密码的验证，应该是用服务器端验证，而不能单单是在客户端用 javascript 验证

  31. 用户名和密码的输入框，应该屏蔽 SQL 注入攻击

  32. 用户名和密码的的输入框，应该禁止输入脚本（防止 XSS 攻击）

  33. 错误登陆的次数限制（防止暴力破解）

  34. 考虑是否支持多用户在同一机器上登录

  35. 考虑一用户在多台机器上登录
    性能测试

  36. 单用户登录系统的响应时间是否符合”3-5-8”原则

  37. 用户数在临界点时并发登录是否还能符合”3-5-8”原则

  38. 压力：大量并发用户登录，系统的响应时间是多少？系统会出现宕机、内存泄露、cpu 饱和、无法登录吗?

  39. 稳定性：系统能否处理并发用户数在临界点以内连续登录 N 个时的场景？
    页面展示测试

  40. 登录页面显示是否正常？文字和图片能否正常显示，相应的提示信息是否正确，按钮的设置和排列是否正常，页面是否简洁壮观等。

  41. 页面默认焦点是否定位在用户名的输入框中

  42. 首次登录时相应的输入框是否为空？或者如果有默认文案，当点击输入框时默认方案是否消失？

  43. 相应的按钮如登录、重置等，是否可用;页面的前进、后退、刷新按钮是否可用？

  44. 快捷键 Tab,Esc,Enter 等，能否控制使用？

  45. 兼容性测试：不同浏览器，不同操作系统，不同分辨率下界面是否正常。
    兼容性测试
    1.不同浏览器下，验证登录页面的显示以及功能正确性；
    2.相同浏览器的不同版本下，验证登录页面的显示以及功能正确性；
    3.不同移动设备终端的不同浏览器下，验证登录页面的显示以及功能正确性；
    4.不同分辨率的界面下，验证登录页面的显示以及功能正确性。
    弱网测试
    1.针对不同的场景使用不同的测试方法，web 可以使用浏览器自带的，app 可以使用 代理等；

微信拜年红包测试用例

功能测试

  1. 红包最多可以输入的金额；
  2. 红包一次性可以发送的最大个数
  3. 在输入红包的钱数和个数时只能输入数字
  4. 当余额不足时，红包发送失败
  5. 发送的红包自己是否可以领取
  6. 发送的红包别人是否可以领取
  7. 红包超过 24 小时是否可以领取
  8. 红包超时未领取，是否退回原账户
  9. 是否可以自己选择支付方式
  10. 红包描述可以输入的最大字符
  11. 余额不足时，是否可以切换支付方式
UI 界面测试
  1. 输入界面是否清晰可见
  2. 红包界面颜色搭配是否美观
  3. 输入金额界面是否有错别字

性能测试

  1. 发红包成功后的跳转时间
  2. 红包超时未领取后的退款时间
  3. 网速较差时，发红包的时间

安全测试

  1. 红包发送成功后，微信是否会收到通知
  2. 包发送失败，余额不会产生变化

弱网测试

  1. 弱网下是否可以发送红包
  2. 弱网下发送红包的时间

易用性测试

  1. 是否可以选择默认支付方式
  2. 余额不足时，是否可以切换支付方式
  3. 是否支持密码支付和指纹支付
